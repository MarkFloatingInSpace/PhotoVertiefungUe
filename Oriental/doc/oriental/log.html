log¶

Send log messages to screen and/or file.

Emit log messages using an instance of Logger.

All instances of Logger send their messages to an application-wide logging kernel singleton. 2 sinks are attached to this kernel:

  • the screen, and

  • the XML log file.

Call selectSinks() to choose a subset of these sinks, to which the kernel will forward its messages.

Set the XML log file’s path using setLogFileName().

setFileMinSeverity() and setScreenMinSeverity() set the application-wide minimum Severitys of log messages. Only log messages whose severity is higher or equal to this threshold will actually be forwared to sreen and/or file.

clearLogFileName()¶

useful for multiprocessing, if child processes’ file logging sink shall be deactivated.

Deactivation of logging to file in sub-processes is generally needed, because log does not support access to the same XML-log by multiple processes! There are 2 work-arounds: - suppress output to file by setting an empty log file name - log to arbitrary, temporary files. - cleanest would be to collect and return the log messages from sub-processes, and then log them in the main process.

There is no robust way to determine if a process is a child process created by multiprocessing. However, if child processes are created with no explicit name, then their names match the following:

if re.match(r'Process-\d+', multiprocessing.current_process().name):
    log.setLogFileName( '' )

Even though this not-so-robust check could be done in every oriental function (mainly plotting), we better initialise multiprocessing.Pool with a process-initializer-function as argument

multiprocessing.Pool tries to pickle its initializer and initargs, which is impossible for boost.Python functions. Thus, we cannot init Pool as Pool( initializer=log.setLogFileName, initargs=”” ), but define here a function implemented in Python to do the job.

usage:

with multiprocessing.Pool( initializer=log.clearLogFileName ) as pool:
    pass
suppressCustomerClearLogFileName()¶

multiprocessing.Pool expects its initializer function to be pickleable

class Logger((object)arg1, (str)moduleName) → None¶

Bases: Boost.Python.instance

a logger whose messages are marked as originating from moduleName

All methods accept an arbitrary number of positional and keyword arguments. The first argument must be a string. The log message is constructed by feeding any additional arguments to the first argument’s method str.format().

Use methods debug(), verbose(), info(), and warning() to emit log messages to the logging kernel. Using these methods, only the settings of selectSinks(), setScreenMinSeverity(), and setFileMinSeverity()mandate if and where a message is actually output.

Use methods severityScreen to not print messages to the XML log file. Settings of selectSinks() and setScreenMinSeverity() decide if such messages are printed to screen.

Use methods severityFile to not print messages to screen. Settings of selectSinks() and setFileMinSeverity() decide if such messages are printed to the XML log file.

Example:

# create a Logger-instance that marks all messages emitted from it with 'my_module'
logger = Logger('my_module')

# emit a log message with severity 'info'
logger.info('hello')

# log message with placeholder for variable content.
# Log message text results as: 'processing time: 30s'
logger.info('processing time: {}s', 30 )

class Severity¶

Bases: Boost.Python.enum

Importance of a log message

Value

Description

debug

lowest importance - use for program de-bugging only

verbose

low importance - additional informations for users, which however may be distracting and are likely to be turned off

info

standard importance - informations about the progress of a program

warning

high importance - use for reporting weird program states that might lead to unexpected output

error

highest importance - use immediately before program abortion
